# 选品雷达 v3 升级规划
> 基于竞品工具调研（Helium 10 / LinkFox / Exploding Topics）+ 现有 v2 代码分析
> 2026-06-04

## 现状盘点

v2 已有的能力：
- ✅ 亚马逊 UK 4榜单（新品/畅销/心愿/礼物）
- ✅ 8个数据源（Amazon + TikTok + HotUKDeals + Temu + Etsy + YouTube + Google Trends + Reddit）
- ✅ 18维度加权评分
- ✅ 6层过滤漏斗（禁词→品牌→评论→价格→利润率→淡季标记）
- ✅ 品类轮换（17个品类）
- ✅ GitHub Pages 仪表盘（12频道标签页 + 状态跟踪）
- ✅ 飞书 cron 推送

**v2 的核心问题：数据有了，但"信号→决策"的转化链路太弱。** 产品过了滤网直接上评分，没有做趋势背离检测、供给缺口分析、信号置信度这些能帮你做判断的分析层。

---

## 要借鉴什么 & 怎么改

### 一、趋势背离检测（学 Helium 10 Trendster）

**做什么：** 不只看"Google Trends热不热"，而是看 Google 搜索热度和亚马逊 BSR 排名的时间差。

**具体方案：**
```
数据源：Google Trends（AnySearch 已有）+ Amazon BSR 历史排名（新增每日快照）
算法：
  信号A = Google Trends 过去30天搜索量变化率
  信号B = Amazon BSR 过去30天排名变化率（排名下降=销量上升）
  
  背离类型：
  📈 Google涨 + Amazon不动 → "需求先行"窗口（+25分）
  📈 Google涨 + Amazon也涨 → "已起来"（+10分，来不及了）
  📉 Google降 + Amazon还稳 → "即将衰退"（-15分）
  📉 两个都降 → 跳过
```

**改动范围：**
- `sources/google_trends.py` — 改为返回趋势方向（rising/stable/falling）+ 变化幅度
- 新增 `data/history/bsr_history.json` — 每次扫描记录 BSR 排名，30天滚动
- `scoring_engine.py` — 新增 `trend_divergence` 维度

**工程量：** 中等（2-3天），主要是 BSR 历史数据的存储和变化率计算

### 二、信号投票机制（学 Exploding Topics）

**做什么：** 现在 v2 的"多源验证"太粗糙——只数有几个 source tag，不做置信度区分。

**具体方案：**
```
信号强度等级：
  🔴 强信号（≥3个独立信号源同时指向）→ +20分，优先推送
  🟠 中信号（2个信号源）→ +10分
  🟡 弱信号（1个信号源）→ 不加分
  ⚪ 无信号 → -10分（说明产品没有外部需求验证）

信号源独立性：
  Amazon内部信号（新品榜+畅销榜+心愿榜）算1组
  外部信号每个独立计数：TikTok=1, Google Trends=1, HotUKDeals=1...
  关键规则：至少要有1个外部信号，纯Amazon内部不算强信号
```

**改动范围：**
- `scoring_engine.py` — 重写 multi-source 计分逻辑，区分内外部信号
- `run_scan_v2.py` — 标记每个产品的信号来源组

**工程量：** 小（1天）

---

### 三、供给缺口指数（学所有工具的共同缺失——自己补）

**做什么：** 这是所有市面工具都没做的。一个产品 TikTok 火不代表你能做，得看亚马逊上供给够不够。

**具体方案：**
```
供给缺口 = 搜索量需求 / 现有供给强度

搜索量需求 = AnySearch 返回的关键词热度（已有）
供给强度 = 该品类前3页总评论数（新增采集）

计算：
  供需比 = 需求信号强度 / ln(总评论数+1)
  
  供需比 > 5  → 深蓝海（+20分）
  供需比 > 3  → 浅蓝海（+10分）
  供需比 > 1  → 平衡（不加不减）
  供需比 < 1  → 红海（-15分）
```

**改动范围：**
- `sources/amazon_uk.py` — 扫描品类页面时记录该品类总评论数
- 新增 `supply_demand_index.py` — 计算供需比
- `scoring_engine.py` — 新增 `supply_demand` 维度
- Dashboard — 卡片上显示供需比标签（🌊深蓝海 / 💧浅蓝海 / ⚖️平衡 / 🔴红海）

**工程量：** 中等（2天）

---

### 四、加速度筛选（学 Helium 10 Black Box）

**做什么：** 不只看产品当前的绝对数据，看它最近的变化趋势。

**具体方案：**
```
数据：每日快照中的 ASIN 级销量估算（已有 BSR → 可推算）

计算：
  7天加速度 = (近7天平均日销 - 前7天平均日销) / 前7天平均日销
  
  加速度 > 50%  → 🚀 飙升品（+15分）
  加速度 > 20%  → 📈 上升品（+8分）
  加速度 < -20% → 📉 下降品（-10分）
```

**改动范围：**
- `data/history/` — 保留至少14天的每日扫描快照
- 新增 `trend_velocity.py` — 计算ASIN级加速度
- `scoring_engine.py` — 新增 `velocity` 维度
- Dashboard — 卡片上加 🚀📈📉 趋势箭头

**工程量：** 中等（2天），但需要积累14天数据后才能生效

---

### 五、蓝海四维筛选升级（学 LinkFox）

**做什么：** 把 LinkFox 的"高销量+少评论+高评分+FBA"四维组合固化为标准筛选条件。

**v2 现状 vs v3 目标：**

| 维度 | v2 | v3 |
|---|---|---|
| 评论数 | 5-300 | 5-100（收紧，300太宽） |
| 评分 | 未筛选 | ≥4.0（排除退货风险品） |
| FBA卖家数 | 未采集 | 采集并显示（FBA<5家 +10分） |
| 销量趋势 | 未追踪 | 7天加速度（见第四点） |

**改动范围：**
- `scanner.py` — 加评分过滤
- `sources/amazon_uk.py` — 采集 FBA 卖家数（从 BSR 页面解析）
- `config.json` — `max_reviews` 从300降到100，新增 `min_rating: 4.0`

**工程量：** 小（1天）

### 六、1688 一键找货源（学 LinkFox 的供应链整合）

**做什么：** 产品卡片上加"一键搜1688"按钮，同时显示3档采购成本的利润预估。

**具体方案：**
```
按钮：点击打开 1688 搜索页（产品英文名）
URL: https://s.1688.com/selloffer/offer_search.htm?keywords={name}

利润预估3档：
  低档 £0.80 → 利润率 XX%
  中档 £1.20 → 利润率 XX%
  高档 £1.50 → 利润率 XX%
```

**改动范围：**
- `generate_html_v2.py` — 卡片加1688按钮 + 3档利润展示
- `config.json` — `sourcing_tiers: [0.80, 1.20, 1.50]`

**工程量：** 小（0.5天）

## 优先级排序

| 优先级 | 改进项 | 工程量 | 预期效果 |
|---|---|---|---|
| **P0** | 信号投票机制 | 1天 | 过滤噪音，只看强信号产品 |
| **P0** | 蓝海四维筛选 | 1天 | 评分过滤+评论上限收紧，减少误判 |
| **P0** | 1688一键找货源 | 0.5天 | 直接加速找供应商流程 |
| **P1** | 供给缺口指数 | 2天 | 真正的蓝海发现能力 |
| **P1** | 趋势背离检测 | 2-3天 | 提前发现"需求先行"产品 |
| **P2** | 加速度筛选 | 2天 | 需要14天数据积累，长期价值大 |

**建议执行顺序：P0全部 → P1供给缺口 → P1趋势背离 → P2加速度**

P0 改完 = 信号更准 + 筛选更严 + 供应链加速，3.5天搞定
P1 改完 = 蓝海发现 + 趋势预警，再加5天
P2 改完 = 完整的趋势跟踪体系，再加2天（但要等数据积累）

## 不做的事

| 功能 | 原因 |
|---|---|
| TikTok Shop GMV 真实数据 | Helium 10 的 TikTok 工具是给 TikTok 卖家用的，不是给 Amazon 选品用的。而且需要付费订阅。维持现有 AnySearch 方案。 |
| 买 Helium 10 / Jungle Scout | 月费 $49-99，且不聚合多源信号。我们的雷达已经在做它们做不到的事。 |
| UK VPN/代理 | 成本高，且 Movers & Shakers 数据价值不如心愿榜。省掉。 |
| Exploding Topics 订阅 | $99/月，且不 Amazon 专用。我们可以用免费的 Google Trends + AnySearch 替代。 |

```
信号采集层     Amazon 4榜单 + AnySearch 8源24查询 + BSR历史快照
    ↓

    ↓
多维评分       18维度 → 升级到22维度
    ↓

```
